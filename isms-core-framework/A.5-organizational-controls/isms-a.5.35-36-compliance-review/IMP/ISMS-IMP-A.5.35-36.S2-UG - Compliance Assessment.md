### Assessment Overview

**Purpose**

This workbook enables managers and system owners to systematically assess compliance with [Organisation]'s information security policies, topic-specific policies, rules, and standards. It supports the operational compliance verification required by ISO 27001:2022 Control A.5.36.

**Scope**

This assessment covers:
- Policy-by-policy compliance verification
- Control-by-control compliance assessment
- Department self-assessment process
- Non-compliance identification and tracking
- Evidence collection for compliance claims

**What This Assessment Covers**

| Domain | Assessment Focus |
|--------|-----------------|
| Policy Compliance | Adherence to security policies |
| Control Compliance | Implementation status of controls |
| Department Assessment | Team/area self-assessment |
| Non-Compliance | Gap identification and tracking |
| Evidence | Documentation of compliance |

**Control Requirements**

ISO 27001:2022 Control A.5.36 states:
> *"Compliance with the organisation's information security policy, topic-specific policies, rules and standards should be regularly reviewed."*

Key requirements:
- Regular review of compliance (not strictly independent)
- Manager-led compliance verification acceptable
- Documented review results
- Timely corrective action for non-compliance
- Progress tracking on remediation

### Completion Walkthrough

**Step 1: Policy_Compliance Sheet**

Assess compliance with each information security policy:

1. **Policy_ID** - Policy document identifier
2. **Policy_Title** - Policy name
3. **Policy_Version** - Current version in use
4. **Last_Review_Date** - When policy was last reviewed for currency
5. **Compliance_Status** - Compliant / Partial / Non-Compliant / Not Applicable
6. **Score** - Numeric score (1.0, 0.5, 0, N/A)
7. **Key_Requirement_1** - First key requirement from policy
8. **Req_1_Status** - Status of requirement 1
9. **Key_Requirement_2** - Second key requirement
10. **Req_2_Status** - Status of requirement 2
11. **Key_Requirement_3** - Third key requirement
12. **Req_3_Status** - Status of requirement 3
13. **Evidence_Ref** - Reference to supporting evidence
14. **Assessor** - Person completing assessment
15. **Assessment_Date** - Date of assessment
16. **Notes** - Additional observations

**Policies to Assess** (typical list):

| Policy | Focus Areas |
|--------|-------------|
| ISMS-POL-01 (Information Security Policy) | Objectives, scope, principles |
| ISMS-POL-A.5.1-4 (Governance) | Roles, responsibilities, reviews |
| ISMS-POL-A.5.9 (Asset Management) | Inventory, ownership, classification |
| ISMS-POL-A.5.10-11 (Asset Usage) | Acceptable use, return of assets |
| ISMS-POL-A.5.15-18 (Access Control) | Access management, privileges |
| ISMS-POL-A.8.x (Technical Controls) | Endpoint, network, application security |
| [Add all applicable policies] | |

**Step 2: Control_Compliance Sheet**

Verify implementation of specific controls:

1. **Control_ID** - ISO 27001 control reference
2. **Control_Name** - Control title
3. **Control_Objective** - What the control achieves
4. **Implementation_Status** - Implemented / Partially / Not Implemented / Not Applicable
5. **Operating_Effectiveness** - Effective / Partially Effective / Ineffective
6. **Compliance_Score** - Numeric (1.0, 0.75, 0.5, 0.25, 0)
7. **Evidence_of_Implementation** - Description of how control is implemented
8. **Evidence_of_Effectiveness** - How effectiveness is verified
9. **Evidence_Ref** - Reference to supporting evidence
10. **Gap_Description** - If not fully compliant, describe gap
11. **Assessor** - Person completing assessment
12. **Assessment_Date** - Date of assessment

**Assessment Criteria**:

| Score | Implementation | Effectiveness |
|-------|----------------|---------------|
| 1.0 | Fully implemented | Consistently effective |
| 0.75 | Largely implemented | Mostly effective |
| 0.5 | Partially implemented | Inconsistently effective |
| 0.25 | Minimally implemented | Rarely effective |
| 0 | Not implemented | Not effective |

**Step 3: Department_Assessment Sheet**

Enable manager-led self-assessment by department:

1. **Department** - Department or team name
2. **Manager** - Department manager
3. **Assessment_Period** - Period covered (e.g., Q1 2026)
4. **Assessment_Date** - Date completed
5. **Security_Awareness** - Staff awareness level (High/Medium/Low)
6. **Policy_Knowledge** - Understanding of policies (High/Medium/Low)
7. **Training_Compliance** - % completed mandatory training
8. **Incident_Reporting** - Compliance with reporting requirements
9. **Asset_Management** - Compliance with asset procedures
10. **Access_Control** - Compliance with access procedures
11. **Data_Handling** - Compliance with data classification/handling
12. **Physical_Security** - Compliance with physical controls
13. **Overall_Status** - Compliant / Partial / Non-Compliant
14. **Key_Issues** - Main compliance concerns identified
15. **Actions_Planned** - Improvement actions
16. **Notes** - Additional observations

**Manager Responsibilities**:

- Ensure staff have read applicable policies
- Verify attendance at security training
- Check compliance with security procedures
- Report non-compliance promptly
- Take corrective action for issues identified

**Step 4: NonCompliance_Register Sheet**

Track identified non-compliance:

1. **NC_ID** - Non-compliance identifier (NC-YYYY-NNN)
2. **Source** - Where identified (Policy Assessment / Control Assessment / Department Assessment / Audit)
3. **Related_Ref** - Policy_ID or Control_ID affected
4. **Description** - Clear description of non-compliance
5. **Severity** - Critical / High / Medium / Low
6. **Area_Affected** - Department or system affected
7. **Root_Cause** - Why non-compliance occurred
8. **Identified_Date** - Date discovered
9. **Owner** - Person responsible for remediation
10. **Corrective_Action** - Action to address
11. **Target_Date** - Remediation deadline
12. **Status** - Open / In Progress / Remediated / Verified / Risk Accepted
13. **Closure_Date** - When closed
14. **Verification_By** - Who verified closure
15. **Notes** - Additional information

**Severity Classification**:

| Severity | Definition | Target Resolution |
|----------|------------|-------------------|
| Critical | Complete non-compliance with major risk | 30 days |
| High | Significant non-compliance | 60 days |
| Medium | Partial compliance with gaps | 90 days |
| Low | Minor deviation | Next review |

**Step 5: Evidence_Register Sheet**

Document evidence supporting compliance claims:

1. **Evidence_ID** - Unique identifier
2. **Assessment_Type** - Policy / Control / Department
3. **Related_Ref** - Assessment item reference
4. **Evidence_Type** - Document / Screenshot / Report / Record / Other
5. **Description** - Brief description of evidence
6. **Storage_Location** - File path or system reference
7. **Collected_Date** - When evidence gathered
8. **Collected_By** - Who collected it
9. **Retention_Until** - Retention end date

**Step 6: Approval_SignOff Sheet**

Obtain assessment approval:

1. **Assessment_Type** - Policy Compliance / Control Compliance / Department Self-Assessment
2. **Assessment_Period** - Period covered
3. **Prepared_By** - Assessor name
4. **Prepared_Date** - Date completed
5. **Reviewed_By** - Reviewer name (supervisor)
6. **Review_Date** - Date reviewed
7. **Approved_By** - Approver name (CISO/ISM)
8. **Approval_Date** - Date approved
9. **Comments** - Any conditions or notes

### Common Pitfalls

❌ **MISTAKE**: Assuming compliance without verification
✅ **CORRECT**: Collect and review evidence for every compliance claim

❌ **MISTAKE**: Department managers skip self-assessment as "too busy"
✅ **CORRECT**: Self-assessment is mandatory; escalate non-completion

❌ **MISTAKE**: Non-compliance tracked informally without register
✅ **CORRECT**: All non-compliance recorded in formal register

❌ **MISTAKE**: Compliance claims without linked evidence
✅ **CORRECT**: Every compliance status requires evidence reference

❌ **MISTAKE**: Outdated policies assessed without checking currency
✅ **CORRECT**: Verify policy version is current before assessing

❌ **MISTAKE**: Non-compliance with no owner assigned
✅ **CORRECT**: Every non-compliance requires assigned owner

❌ **MISTAKE**: Corrective actions not tracked to completion
✅ **CORRECT**: Track all actions until verified closure

❌ **MISTAKE**: Assessment done once and never updated
✅ **CORRECT**: Assessments refreshed per scheduled frequency

### Review & Approval

**Review Workflow**:

1. Assessor completes assessment sections
2. Evidence collected and referenced
3. Information Security reviews for completeness
4. CISO approves final assessment
5. Non-compliance items tracked to closure

## 1. Purpose and Scope

### 1.1 Purpose

This document establishes the **assessment methodology and evidence framework** for measuring compliance with ISO/IEC 27001:2022 Controls A.8.26 (Application Security Requirements), A.8.25 (Secure Development Lifecycle), and A.8.29 (Security Testing in Development and Acceptance).

### 1.2 Objectives

- **Measure** compliance with security requirements, SDLC integration, and security testing requirements
- **Collect** evidence systematically for internal audits and external certification
- **Track** trends and identify improvement opportunities
- **Demonstrate** ISO 27001:2022 compliance to auditors
- **Enable** data-driven decision making for security investments
- **Support** continuous improvement through objective metrics

### 1.3 Assessment Approach

**Systematic Assessment**:
- **Application-by-application** assessment (not organization-wide generic)
- **Risk-based** (depth of assessment matches application risk classification)
- **Evidence-based** (objective evidence, not subjective opinions)
- **Automated** (Excel workbooks generated by Python scripts for consistency)
- **Periodic** (annual minimum, quarterly for high-risk applications)

**Assessment Components**:
1. **Workbook 1**: Security Requirements Compliance (A.8.26)
2. **Workbook 2**: SDLC Security Activities Compliance (A.8.25)
3. **Workbook 3**: Security Testing Results (A.8.29)
4. **Workbook 4**: Vulnerability Remediation Tracking (A.8.29)
5. **Dashboard**: Secure Development Maturity (aggregates all workbooks)

## 2. Assessment Methodology Overview

### 2.1 Assessment Lifecycle

```
┌─────────────────────────────────────────────────────────────┐
│           Assessment Lifecycle (Annual Cycle)               │
└─────────────────────────────────────────────────────────────┘

Quarter 1: Planning & Data Collection
├─ Identify applications for assessment
├─ Collect evidence (requirements docs, test results, metrics)
├─ Populate assessment workbooks
└─ Initial gap analysis

Quarter 2: Assessment Execution
├─ Complete all assessment workbooks
├─ Generate compliance dashboard
├─ Analyze findings and gaps
└─ Develop remediation plans

Quarter 3: Remediation
├─ Address identified gaps
├─ Implement improvements
├─ Track remediation progress
└─ Update processes and standards

Quarter 4: Review & Continuous Improvement
├─ Measure improvement
├─ Update assessment methodology
├─ Report to management and board
└─ Plan next assessment cycle
```

### 2.2 Assessment Frequency

**High-Risk Applications**:
- **Quarterly assessments** (lightweight, focused on changes)
- **Annual comprehensive assessment** (full workbook completion)

**Medium-Risk Applications**:
- **Semi-annual assessments** (targeted)
- **Annual comprehensive assessment**

**Low-Risk Applications**:
- **Annual assessment** (focused on critical requirements)

**Trigger-Based Assessments** (in addition to scheduled):
- Major application changes (new features, architecture changes)
- Security incidents
- Regulatory changes
- Post-remediation validation

## 3. Evidence Collection Framework

### 3.1 Evidence Types

Organizations **SHALL** collect evidence per control:

**A.8.26 (Security Requirements) Evidence**:
- ✅ Security requirements specification documents (approved, current)
- ✅ Application risk classification decisions
- ✅ Threat modeling documentation
- ✅ Security architecture review reports
- ✅ Requirements traceability matrices
- ✅ Security requirements approval records

**A.8.25 (Secure Development Lifecycle) Evidence**:
- ✅ SDLC documentation showing security activities per phase
- ✅ Code review records (peer review + security-focused review)
- ✅ Security tool deployment evidence (SAST, SCA, secret scanning)
- ✅ Security tool scan results and remediation tracking
- ✅ Developer security training completion records
- ✅ Security defect tracking data (open, remediated, SLA compliance)

**A.8.29 (Security Testing) Evidence**:
- ✅ SAST scan reports and findings remediation
- ✅ DAST scan reports and findings remediation
- ✅ SCA scan reports and dependency remediation
- ✅ Penetration testing reports (if applicable)
- ✅ Security acceptance testing results
- ✅ Vulnerability remediation tracking (SLA compliance)

### 3.2 Evidence Quality Standards

Evidence **SHALL** meet quality standards:

**Objective Evidence**:
- ✅ Documented and verifiable (not verbal claims)
- ✅ Timestamped (when was evidence created/collected?)
- ✅ Traceable (who created it, approved it?)
- ✅ Current (not outdated)

**Sufficient Evidence**:
- ✅ Demonstrates control implementation (not just intent)
- ✅ Representative (covers meaningful sample of applications)
- ✅ Complete (addresses all requirements)

**Reliable Evidence**:
- ✅ From authoritative sources (official systems, approved documents)
- ✅ Consistent across multiple sources
- ✅ Independently verifiable

### 3.3 Evidence Collection Process

Organizations **SHALL**:

**Centralize evidence** in evidence repository:
- Document management system (SharePoint, Confluence, shared drive)
- Security tool repositories (SAST/DAST/SCA platforms)
- Defect tracking systems (Jira, Azure DevOps, etc.)
- Code repository metadata (GitHub, GitLab)

**Maintain evidence chain**:
- Version control for documents
- Audit trail for approvals
- Retention per regulatory requirements (typically 3-7 years)

**Protect evidence**:
- Access controls (only authorized personnel)
- Encryption for sensitive evidence
- Backup and recovery

## 4. Assessment Workbook Framework

### 4.1 Workbook Overview

Assessment workbooks are **Excel-based tools** generated by Python scripts to ensure:
- **Consistency**: Same structure and formulas across all assessments
- **Automation**: Auto-calculation of scores and compliance percentages
- **Traceability**: Links to evidence sources
- **Reporting**: Dashboard visualizations and export capabilities

All workbooks follow common structure:
1. **Instructions & Legend** sheet
2. **Assessment data** sheets (application-specific)
3. **Summary/Dashboard** sheet (aggregated results)
4. **Evidence Register** sheet (links to evidence)
5. **Approval/Sign-Off** sheet

### 4.2 Workbook 1: Security Requirements Compliance (A.8.26)

**Purpose**: Assess completeness and quality of security requirements for each application

**Assessment Sheets**:

**Application Inventory**:
- Application name, owner, business criticality
- Risk classification (High/Medium/Low)
- Technology stack (languages, frameworks, platforms)
- Regulatory scope (GDPR, PCI DSS, HIPAA, etc.)

**Security Requirements Documentation Status**:
- Requirements document exists? (Yes/No)
- Requirements approved? (Yes/No)
- Requirements current? (Last updated, review date)
- Requirements completeness score (0-100%)
  - Functional requirements documented
  - Non-functional requirements documented
  - Data protection requirements documented
  - Compliance requirements documented

**Threat Modeling Status**:
- Threat model exists? (Yes/No)
- Threat model complete? (Initial/Complete/Current)
- Threat modeling methodology used (STRIDE, PASTA, etc.)
- Last updated date
- Identified threats count
- Mitigated threats count

**Security Architecture Review**:
- Architecture review conducted? (Yes/No)
- Reviewer (Security Architect, Security Team, External)
- Review date
- Review findings (count by severity)
- Critical findings resolved? (Yes/No)
- Architecture approved? (Yes/No)

**Requirements Traceability Matrix**:
- Traceability matrix exists? (Yes/No)
- Requirements linked to design? (%)
- Requirements linked to code? (%)
- Requirements linked to tests? (%)
- Overall traceability score (0-100%)

**Compliance Scoring**:
- **Per-application score** (0-100%): Weighted average of all criteria
- **Portfolio score**: Average across all applications (weighted by risk)

### 4.3 Workbook 2: SDLC Security Activities Compliance (A.8.25)

**Purpose**: Track security activities executed in each SDLC phase per application

**Assessment Sheets**:

**SDLC Phase Security Activities**:
- For each application and SDLC phase (Requirements, Design, Development, Testing, Deployment, Maintenance):
  - Planned security activities (checkboxes)
  - Executed security activities (checkboxes)
  - Completion % per phase
  - Evidence links

**Secure Coding Standard Compliance**:
- Secure coding standards documented? (Yes/No, link to standards)
- Developers trained on standards? (% trained)
- Standards compliance checked in code review? (Yes/No)

**Code Review Metrics**:
- Code review required? (Yes/No - based on policy)
- Peer code review coverage (% of code reviewed)
- Security-focused code review for high-risk code? (Yes/No)
- Average review turnaround time (days)
- Code review findings (count, severity breakdown)

**Security Tools Deployment**:
- SAST deployed? (Yes/No/Planned/N/A)
- SCA deployed? (Yes/No/Planned/N/A)
- Secret scanning deployed? (Yes/No/Planned/N/A)
- IDE security plugins recommended? (Yes/No)
- CI/CD security integration? (Yes/No/Partial)

**Security Tools Usage**:
- SAST scan frequency (Per commit/Daily/Weekly/None)
- SCA scan frequency (Daily/Weekly/None)
- Last SAST scan date
- Last SCA scan date
- Tool scan coverage (% of projects)

**Developer Security Training**:
- Initial security training completion (% of developers)
- Annual refresher training completion (% of developers)
- Security champion assigned to team? (Yes/No)
- Security champion trained? (Yes/No)

**Security Defect Management**:
- Security defects tracked in defect system? (Yes/No)
- Open security defects (count by severity)
- Overdue security defects (past SLA)
- Mean time to remediate (days, by severity)
- SLA compliance rate (%)

**Compliance Scoring**:
- **Per-application score** (0-100%): Weighted average of SDLC activities, tools, training, defect management
- **Portfolio score**: Average across applications

### 4.4 Workbook 3: Security Testing Results (A.8.29)

**Purpose**: Track security testing execution and findings per application

**Assessment Sheets**:

**Security Testing Coverage**:
- SAST testing performed? (Yes/No)
- DAST testing performed? (Yes/No)
- SCA testing performed? (Yes/No)
- IAST testing performed? (Yes/No/N/A)
- Penetration testing performed? (Yes/No/Planned)
- Security acceptance testing performed? (Yes/No)
- Testing coverage score (0-100% based on required tests for risk level)

**SAST Scan Results**:
- Last SAST scan date
- Scan status (Passed/Failed/Warning)
- Findings by severity (Critical/High/Medium/Low counts)
- True positive rate (estimated %)
- False positives suppressed (count, documented?)
- Remediation rate (% of findings fixed)

**DAST Scan Results**:
- Last DAST scan date
- Scan type (Authenticated/Unauthenticated/Both)
- Scan status (Passed/Failed/Warning)
- Findings by severity (Critical/High/Medium/Low counts)
- Remediation rate

**SCA Scan Results**:
- Last SCA scan date
- Vulnerable dependencies (count by severity)
- Outdated dependencies (count)
- License compliance issues (count)
- Remediation rate

**Penetration Testing Results** (if applicable):
- Last penetration test date
- Test type (Black box/Gray box/White box)
- Tester (Internal/External)
- Findings by severity
- Retesting performed? (Yes/No)
- All findings remediated? (Yes/No/In Progress)

**Security Acceptance Testing**:
- Security test cases defined? (Yes/No, count)
- Security test cases executed? (%, count)
- Security test cases passed? (%, count)
- Security sign-off obtained? (Yes/No, date)

**Compliance Scoring**:
- **Per-application score** (0-100%): Based on testing coverage, finding remediation, sign-off
- **Portfolio score**: Average across applications

### 4.5 Workbook 4: Vulnerability Remediation Tracking (A.8.29)

**Purpose**: Track open vulnerabilities and remediation compliance

**Assessment Sheets**:

**Open Vulnerabilities Inventory**:
- Vulnerability ID
- Source (SAST/DAST/SCA/Pentest)
- Application
- Severity (Critical/High/Medium/Low)
- Vulnerability type (Injection, XSS, Authentication, etc.)
- Discovery date
- SLA due date
- Status (Open/In Progress/Resolved/Accepted Risk)
- Assigned to
- Age (days open)
- Overdue? (Yes/No)

**Remediation SLA Compliance**:
- Total vulnerabilities by severity
- Overdue vulnerabilities by severity
- SLA compliance rate (%)
- Average remediation time by severity (vs. SLA target)

**Vulnerability Trends**:
- Vulnerabilities opened per month (by severity)
- Vulnerabilities closed per month (by severity)
- Net change (increasing/decreasing backlog)
- Backlog age distribution (0-30 days, 31-60 days, 61-90 days, 90+ days)

**Security Technical Debt**:
- Total technical debt items
- High-priority technical debt
- Technical debt with remediation plans (Yes/No)
- Technical debt review date (last reviewed)

**Compliance Scoring**:
- **SLA compliance score** (0-100%): % of vulnerabilities remediated within SLA
- **Backlog health score** (0-100%): Based on backlog size, age, and trends
- **Overall remediation score**: Weighted average

### 4.6 Dashboard: Secure Development Maturity

**Purpose**: Aggregate all assessment workbooks into executive dashboard

**Dashboard Components**:

**Overall Secure Development Maturity Score** (0-100%):
- Weighted average of:
  - Security Requirements Compliance (A.8.26) - 30%
  - SDLC Security Activities (A.8.25) - 35%
  - Security Testing Coverage (A.8.29) - 25%
  - Vulnerability Remediation (A.8.29) - 10%

**Control-Specific Scores**:
- A.8.26 Security Requirements: XX%
- A.8.25 Secure Development: XX%
- A.8.29 Security Testing: XX%

**Maturity Level Classification**:
- **Level 1 - Initial** (0-40%): Ad-hoc security, minimal requirements
- **Level 2 - Developing** (41-60%): Basic security practices, some automation
- **Level 3 - Defined** (61-80%): Documented processes, consistent execution
- **Level 4 - Managed** (81-90%): Metrics-driven, proactive security
- **Level 5 - Optimizing** (91-100%): Continuous improvement, industry-leading

**Trend Analysis**:
- Maturity score over time (quarterly/annual trends)
- Improvement or regression indicators
- Forecast to target maturity level

**Risk Heatmap**:
- Applications plotted by risk level vs. compliance score
- Identify high-risk, low-compliance applications (priority remediation)

**Key Findings**:
- Top 5 gaps/weaknesses
- Top 5 strengths
- Critical action items
- Quick wins (high impact, low effort)

**Recommendations**:
- Prioritized remediation roadmap
- Resource allocation suggestions
- Process improvement opportunities

## 5. Compliance Scoring Methodology

### 5.1 Scoring Principles

**Objective Scoring**:
- Based on measurable evidence (not subjective opinion)
- Automated calculation (Excel formulas)
- Consistent across applications

**Risk-Weighted Scoring**:
- High-risk applications have stricter pass criteria
- Low-risk applications have relaxed criteria
- Portfolio scores weighted by application risk

**Pass/Fail Thresholds**:
- **Compliant**: ≥ 80% score
- **Partially Compliant**: 60-79% score
- **Non-Compliant**: < 60% score

### 5.2 Calculation Examples

**Workbook 1 - Security Requirements Score**:
```
Requirements Score = (
  Requirements Documentation (25%) +
  Threat Modeling (25%) +
  Architecture Review (25%) +
  Traceability (25%)
) × Risk Factor

Risk Factor:
- High Risk: 1.0 (no reduction, full requirements expected)
- Medium Risk: 0.9 (slight adjustment)
- Low Risk: 0.8 (adjusted expectations)
```

**Workbook 2 - SDLC Activities Score**:
```
SDLC Score = (
  SDLC Phase Activities (30%) +
  Code Review (20%) +
  Security Tools (25%) +
  Developer Training (15%) +
  Defect Management (10%)
)
```

**Overall Maturity Score**:
```
Maturity = (
  Requirements (30%) +
  SDLC Activities (35%) +
  Testing Coverage (25%) +
  Vulnerability Remediation (10%)
)
```

## 6. Continuous Monitoring and Improvement

### 6.1 Continuous Monitoring

Organizations **SHOULD** implement continuous monitoring:

**Real-Time Metrics**:
- SAST/DAST/SCA scan results (automated, continuous)
- Open vulnerability counts (updated daily)
- SLA breach alerts (automated notifications)

**Dashboard Refresh**:
- **Weekly**: Vulnerability metrics updated
- **Monthly**: SDLC activity metrics updated
- **Quarterly**: Full assessment refresh for high-risk apps
- **Annually**: Comprehensive assessment for all apps

### 6.2 Improvement Process

Organizations **SHALL** use assessment results for improvement:

**Quarterly Reviews**:
- Review dashboard with development management
- Identify top gaps and remediation priorities
- Allocate resources for remediation
- Track improvement progress

**Annual Reviews**:
- Comprehensive review with executive management
- Report to board (ISO 27001 compliance status)
- Update secure development strategy
- Set targets for next year

**Continuous Improvement**:
- Update processes based on findings
- Enhance training based on common gaps
- Improve tools and automation
- Share best practices across teams

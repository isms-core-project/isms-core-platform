## 1. Purpose and Scope

### 1.1 Purpose

This document provides **SOC operational procedures** for handling security alerts from detection to resolution. These procedures ensure:

- **Consistent triage** - Every alert handled with same rigor
- **Effective investigation** - Step-by-step playbooks for common scenarios
- **Timely escalation** - Clear criteria and paths for escalation
- **Evidence preservation** - Forensic readiness from first response
- **Continuous improvement** - Documented false positives enable tuning

**This is your SOC desk reference. Keep it handy.**

### 1.2 Scope

Procedures apply to **all alerts** generated by monitoring infrastructure:

- SIEM correlation rules
- IDS/IPS signatures
- EDR/AV detections
- Baseline deviation alerts
- Threat intelligence matches
- Manual escalations from users/IT

**Out of Scope**: Incident response after escalation (see IR playbooks). This document covers **alert handling through initial triage/investigation**.

### 1.3 Using This Document

**For Daily Operations**: Follow Section 2 (Triage) for every alert, then Section 3 (Playbooks) for specific alert types.

**For Training**: New analysts study Section 2-3, practice with historical alerts, shadow experienced analysts.

**For Continuous Improvement**: Document false positives in Section 6, update playbooks quarterly.

## 2. Alert Triage Procedure (EVERY ALERT FOLLOWS THIS)

### 2.1 Overview

**Triage = Rapid assessment to determine alert disposition.**

**Possible Dispositions**:
- ✅ **True Positive** → Escalate to investigation/incident response
- ❌ **False Positive** → Document scenario, close, tune if recurring
- ⚠️ **Benign True Positive** → Real event, but non-malicious (close with notes)
- 🔍 **Requires Investigation** → Not immediately obvious, needs deeper analysis

**Triage Timeframes** (from ISMS-POL-A.8.16-S2.3):
- **Critical (P1)**: <15 minutes
- **High (P2)**: <1 hour
- **Medium (P3)**: <4 hours
- **Low (P4)**: <1 business day

---

### 2.2 Step-by-Step Triage Process

#### Step 1: Acknowledge Alert

**Actions**:
- [ ] Claim alert in SIEM/ticketing system (assign to yourself)
- [ ] Update status: NEW → INVESTIGATING
- [ ] Note start time (for SLA tracking)

**Purpose**: Prevents duplicate triage, establishes accountability.

---

#### Step 2: Read Alert

**What to Look For**:
- [ ] **Alert Name/Title** - What is being detected?
- [ ] **Alert Severity** - Critical / High / Medium / Low
- [ ] **Triggered Time** - When did event occur?
- [ ] **Key Fields** - Source IP, destination IP, user, hostname, process, etc.
- [ ] **Alert Description** - What does this alert mean? (Should reference MITRE ATT&CK if available)
- [ ] **Enrichment Data** - Threat intel matches, asset context, user info

**Example Alert**:
```
ALERT: Brute Force Authentication Detected
Severity: High (P2)
Time: [Approval Date] 14:32:15
Source IP: 203.0.113.45
Target: PROD-DC-01.company.local
User: admin
Event Count: 87 failed logins in 15 minutes
Enrichment: Source IP = known botnet (AlienVault OTX)
MITRE: T1110.001 - Password Guessing
```

---

#### Step 3: Gather Context

**Questions to Answer**:

**About the User** (if user-related):
- [ ] Is this a real user or service account?
- [ ] What's their role/department?
- [ ] Do they normally access this system?
- [ ] Any recent tickets (password reset, access request)?

**About the Asset** (if asset-related):
- [ ] What's the system criticality? (Critical/High/Medium/Low)
- [ ] What's the system function? (DC, DB, web server, workstation)
- [ ] Who owns this system?
- [ ] Is system in maintenance window?

**About the Source** (if external):
- [ ] Is source IP internal or external?
- [ ] If external: Geo-location, reputation (threat intel)
- [ ] Any recent communications with this IP?

**About Recent Activity**:
- [ ] Any related alerts in past 24 hours? (same user, same IP, same system)
- [ ] Any successful authentication after failed attempts?
- [ ] Any file access, process execution, network connections following this event?

**Where to Find Context**:
- SIEM user/asset enrichment
- CMDB / Asset Management System
- Active Directory (user info)
- Threat Intelligence Platform
- Recent alert history
- Ticketing system (recent user tickets)

---

#### Step 4: Determine Disposition

**Decision Tree**:
```
Is this event expected/authorized?
├─ YES → Benign True Positive
│  ├─ Document scenario
│  ├─ Close alert
│  └─ Consider whitelisting if recurring
│
└─ NO → Continue...
    │
    Is there clear malicious intent or impact?
    ├─ YES → True Positive
    │  ├─ Escalate to IR immediately
    │  └─ Follow playbook
    │
    └─ NO → Continue...
        │
        Is the alert technically accurate but non-threatening?
        ├─ YES → False Positive
        │  ├─ Document why FP
        │  ├─ Close alert
        │  └─ Tune detection if recurring
        │
        └─ UNCLEAR → Requires Investigation
            └─ Proceed to Section 3 (Playbooks)
```

**Examples by Disposition**:

**True Positive**:
- Brute force followed by successful login from known botnet IP
- Malware detected on endpoint with C2 communication
- Data exfiltration to unauthorized cloud storage
- Lateral movement detected (admin login to unrelated systems)

**False Positive**:
- Security scanner triggered IDS signature (authorized scan)
- Failed logins = user typo'd password 3 times
- Baseline deviation = legitimate business event (month-end processing)

**Benign True Positive**:
- Failed logins after password reset (user entering old password)
- Backup job triggered data transfer alert (scheduled, authorized)
- IT admin accessing multiple systems (legitimate administrative activity)

**Requires Investigation**:
- Failed logins from unusual location (employee traveling?)
- Process execution on server (legitimate software update or malware?)
- Large file transfer (business file share or data theft?)

---

#### Step 5: Document Triage

**Minimum Documentation** (in alert ticket):
```
TRIAGE SUMMARY
--------------
Analyst: [Your Name]
Triage Start: [DD.MM.YYYY HH:MM]
Triage End: [DD.MM.YYYY HH:MM]
Disposition: [True Positive / False Positive / Benign TP / Requires Investigation]

FINDINGS:
[Concise summary of what you found during triage]

RATIONALE:
[Why did you reach this disposition?]

ACTIONS TAKEN:
[ ] Reviewed alert details
[ ] Checked user/asset context
[ ] Reviewed recent activity
[ ] [Other actions]

NEXT STEPS:
[If escalating: Brief investigation plan]
[If closing: Recommendation for tuning]
```

**Example**:
```
TRIAGE SUMMARY
Analyst: Jane Smith
Triage Start: [Approval Date] 14:35
Triage End: [Approval Date] 14:42
Disposition: False Positive

FINDINGS:
Alert triggered for 87 failed logins to PROD-DC-01 from 203.0.113.45. 
Source IP identified as authorized security scanner (Qualys) performing 
authenticated scan. Scan window: 14:30-15:00 as per monthly schedule.

RATIONALE:
Authorized activity, scanner credentials intentionally fail during 
vulnerability assessment (testing account lockout controls).

ACTIONS TAKEN:
[X] Reviewed alert details
[X] Checked source IP (Qualys scanner confirmed)
[X] Verified scan schedule (monthly scan, expected)
[X] Confirmed no successful logins from this IP

NEXT STEPS:
Whitelist Qualys scanner IP for this specific alert type to prevent 
monthly false positives. Created tuning ticket #SOC-2847.
```

### 2.3 Triage Quality Standards

**Every Triage MUST Include**:
- [ ] Analyst name and timestamps
- [ ] Clear disposition with rationale
- [ ] Documented investigation steps
- [ ] Next steps (escalation or tuning)

**Anti-Cargo-Cult Triage**:
❌ "Reviewed, looks fine, closing" (no evidence of actual review)  
✅ "Checked user dept (Finance), normal login location (HQ), no recent alerts, closing as user password typo"

❌ "Escalated to IR" (no context provided)  
✅ "Escalated: Confirmed malware with C2 beacon to 1.2.3.4, host isolated, IR notified at 14:45"

## 3. Investigation Playbooks by Alert Type

### 3.1 Playbook: Brute Force Authentication

**Alert Signature**: Failed authentication events exceeding threshold

**MITRE ATT&CK**: T1110 - Brute Force (Password Guessing, Password Spraying)

#### Investigation Steps

**Step 1: Confirm Failed Attempts**
- [ ] Review failed login events (time range, count)
- [ ] Identify: Source IP, target system, username(s) attempted
- [ ] Pattern: Single username (guessing) or multiple usernames (spraying)?

**Step 2: Check for Successful Login**
- [ ] Did any successful login occur from same source IP?
- [ ] If YES → Critical escalation (attacker gained access)
- [ ] Timeframe: Within 1 hour of failed attempts?

**Step 3: Analyze Source**
- [ ] Source IP internal or external?
- [ ] If external: Check threat intel (known botnet, Tor exit node?)
- [ ] If internal: Which workstation/server? Who has access?
- [ ] Geographic location (does it match user's normal location?)

**Step 4: Check User Account**
- [ ] Real user or service account?
- [ ] Is account locked? (If not, why? Account lockout policy working?)
- [ ] Contact user: "Did you attempt to login and have trouble?"
- [ ] Recent password change? (User might be using old password)

**Step 5: Review Recent Activity**
- [ ] Any other authentication activity from this user/IP in past 24 hours?
- [ ] Any alerts on related systems?

#### Evidence to Collect
- [ ] Authentication logs (failed + successful) ±2 hours around event
- [ ] Source IP geo-location and reputation
- [ ] User account details (last password change, lockout status)
- [ ] Network connection logs (did source IP connect to other systems?)

#### Escalation Criteria

**Escalate Immediately if**:
- ✅ Successful login occurred after failed attempts
- ✅ Source IP is known attack infrastructure (botnet, Tor)
- ✅ Multiple accounts targeted (password spraying)
- ✅ Privileged account targeted (admin, service account)
- ✅ Source is internal (possible compromised workstation)

**Close as FP/Benign if**:
- ❌ User confirms legitimate attempts (forgot password)
- ❌ Source is authorized system (monitoring tool, SSO health check)
- ❌ Account lockout worked correctly, no successful login
- ❌ Password reset occurred just before attempts (user using old password)

---

### 3.2 Playbook: Lateral Movement

**Alert Signature**: Administrative login to multiple systems in short time

**MITRE ATT&CK**: T1021 - Remote Services (RDP, SSH, SMB, WinRM)

#### Investigation Steps

**Step 1: Identify Movement Pattern**
- [ ] Which user/account?
- [ ] Which systems accessed? (List all)
- [ ] Timeframe of accesses?
- [ ] Login method (RDP, SSH, SMB, WinRM)?

**Step 2: Validate User Legitimacy**
- [ ] Is this user's normal behavior? (Check historical patterns)
- [ ] User's role: Should they have admin access to these systems?
- [ ] Are accessed systems related? (e.g., all DB servers) or unrelated?
- [ ] Contact user: "Are you currently accessing these systems?"

**Step 3: Check Source**
- [ ] Source IP/workstation: Where did logins originate?
- [ ] Is source system itself compromised?
- [ ] Any malware alerts on source system?

**Step 4: Review Activity on Target Systems**
- [ ] After login, what processes were executed?
- [ ] Any file access (especially sensitive files)?
- [ ] Any new services/scheduled tasks created?
- [ ] Any credential dumping activity (mimikatz, lsass access)?

**Step 5: Timeline Reconstruction**
```
14:25 - Admin login to SERVER-01 from 10.1.2.50
14:28 - Admin login to SERVER-02 from 10.1.2.50
14:31 - Admin login to SERVER-03 from 10.1.2.50
14:33 - Process execution: powershell.exe on all three servers
```
Fast, sequential, automated = suspicious.

#### Evidence to Collect
- [ ] RDP/SSH connection logs for all accessed systems
- [ ] Process execution logs (Windows Event 4688, Sysmon, EDR)
- [ ] File access logs (especially for sensitive directories)
- [ ] Network connection logs (lateral connections between systems)
- [ ] Credential access logs (lsass dumps, registry access)

#### Escalation Criteria

**Escalate Immediately if**:
- ✅ User denies activity OR is unreachable
- ✅ Accessed systems are unrelated to user's role
- ✅ Process execution includes known attack tools (mimikatz, psexec, etc.)
- ✅ Credential dumping activity detected
- ✅ Activity pattern is automated (too fast for human)
- ✅ Activity outside business hours for this user

**Close as Benign if**:
- ❌ User confirms legitimate administrative activity
- ❌ Systems accessed are within user's responsibility
- ❌ Activity matches known maintenance window
- ❌ Process execution is legitimate (patch deployment, monitoring)

---

### 3.3 Playbook: Data Exfiltration

**Alert Signature**: Large data transfer to external destination

**MITRE ATT&CK**: T1041 - Exfiltration Over C2 Channel, T1048 - Exfiltration Over Alternative Protocol

#### Investigation Steps

**Step 1: Quantify Transfer**
- [ ] Volume transferred (MB/GB)
- [ ] Destination IP/domain
- [ ] Protocol (HTTP/HTTPS, FTP, DNS tunneling, cloud storage API)
- [ ] Source: Which system initiated transfer?
- [ ] User context (if available)

**Step 2: Identify Destination**
- [ ] External IP/domain: What is it?
- [ ] Geo-location of destination
- [ ] Known service? (Google Drive, Dropbox, personal email, unknown)
- [ ] Threat intel: Is destination known malicious?

**Step 3: Determine Data Type**
- [ ] Can you identify what was transferred?
- [ ] File access logs: Which files were read before transfer?
- [ ] If cloud storage: Can you see file names? (depends on encryption)
- [ ] Data sensitivity: Customer data, financial data, IP, or benign?

**Step 4: Check Authorization**
- [ ] Is this user authorized to transfer data externally?
- [ ] Business justification: Why would user need to send data to this destination?
- [ ] Contact user: "Did you upload files to [destination]?"
- [ ] Contact manager: "Is this user authorized to export data?"

**Step 5: Review Broader Activity**
- [ ] Any other exfiltration from same user/system?
- [ ] Any malware alerts on source system?
- [ ] Any credential theft preceding this?

#### Evidence to Collect
- [ ] Network flow logs (src, dst, bytes transferred, protocol)
- [ ] Proxy logs (if HTTPS, at least domain/URL)
- [ ] File access logs on source system
- [ ] User activity logs (authentication, application usage)
- [ ] Destination analysis (whois, threat intel, service identification)

#### Escalation Criteria

**Escalate Immediately if**:
- ✅ Destination is unknown/suspicious (not authorized cloud service)
- ✅ Data is sensitive (customer PII, financial, IP)
- ✅ User denies activity OR cannot provide business justification
- ✅ Transfer volume is extreme (>10 GB)
- ✅ Destination has threat intel indicators (C2, known malicious)
- ✅ Transfer occurred after-hours or from compromised account

**Close as Benign if**:
- ❌ User confirms legitimate file sharing (to partner, customer, etc.)
- ❌ Destination is authorized cloud service (OneDrive, corporate Dropbox)
- ❌ Data is non-sensitive (public documentation, marketing materials)
- ❌ Transfer volume consistent with business activity

### 3.4 Playbook: Malware Detection

**Alert Signature**: EDR/AV/IDS detects malware signature or behavior

**MITRE ATT&CK**: T1204 - User Execution, T1059 - Command and Scripting Interpreter

#### Investigation Steps

**Step 1: Identify Malware**
- [ ] Malware name/signature (e.g., "Trojan.Win32.Generic")
- [ ] File path and hash (MD5, SHA256)
- [ ] Detection method (signature, heuristic, behavior)
- [ ] Affected system and user

**Step 2: Check Containment Status**
- [ ] Did EDR/AV quarantine the file?
- [ ] Is host isolated from network?
- [ ] Is malware still running? (Check process list)
- [ ] If not contained: **Isolate immediately, then continue investigation**

**Step 3: Analyze Malware Behavior**
- [ ] What did malware attempt to do?
  - Network connections (C2 communication?)
  - File encryption (ransomware?)
  - Credential access (info stealer?)
  - Persistence mechanisms (registry, scheduled tasks?)
- [ ] Check EDR telemetry for full behavior timeline

**Step 4: Determine Infection Vector**
- [ ] How did malware arrive?
  - Email attachment (check email logs)
  - Web download (check proxy logs)
  - USB drive (check device logs)
  - Lateral movement (check authentication logs)
- [ ] User activity: What was user doing before infection?

**Step 5: Check for Lateral Spread**
- [ ] Any other systems communicating with this host?
- [ ] Any similar alerts on other systems?
- [ ] Did malware attempt network propagation (SMB, RDP)?

#### Evidence to Collect
- [ ] Malware sample (from quarantine)
- [ ] Process execution timeline (EDR telemetry)
- [ ] Network connections from infected host
- [ ] File system changes (new files, modified files)
- [ ] Registry changes (persistence mechanisms)
- [ ] Email logs (if phishing vector)

#### Escalation Criteria

**Escalate Immediately** (ALL malware detections escalate):
- ✅ Any confirmed malware detection requires IR involvement
- ✅ Prioritize based on:
  - Did malware communicate with C2? (High priority)
  - Is host contained? (If not, immediate escalation)
  - Is malware ransomware? (Critical priority)
  - Did malware spread? (Critical priority)

**Close as FP Only if**:
- ❌ File is confirmed legitimate (vendor-signed, known application)
- ❌ Detection is test malware (EICAR test file, authorized testing)
- ❌ Heuristic false positive on benign software (verify carefully!)

**Note**: Err on side of escalation. False negatives in malware detection = breach.

### 3.5 Playbook: Baseline Deviation

**Alert Signature**: Metric exceeds established baseline threshold

**MITRE ATT&CK**: Varies (depends on metric deviating)

#### Investigation Steps

**Step 1: Review Baseline Context**
- [ ] Which metric deviated? (e.g., authentication events, network traffic)
- [ ] Baseline value vs. observed value (how much deviation?)
- [ ] Time of deviation (business hours, off-hours, weekend?)
- [ ] System affected

**Step 2: Check for Legitimate Business Events**
- [ ] Is there a business cycle explanation?
  - Month-end processing
  - Quarterly reporting
  - Annual events (open enrollment, tax season)
  - Marketing campaigns (website traffic spike)
- [ ] Any planned activities?
  - System migrations
  - User onboarding
  - Software deployments
- [ ] Contact system owner: "Any unusual but legitimate activity?"

**Step 3: Compare to Historical Patterns**
- [ ] Has this happened before? (Check past 90 days)
- [ ] If recurring, is it on predictable schedule? (e.g., first Monday of month)
- [ ] If first occurrence, what changed?

**Step 4: Investigate Underlying Events**
- [ ] Drill into specific logs that comprise the deviated metric
- [ ] Are events from expected sources/users?
- [ ] Any patterns? (e.g., all events from single IP, single user)

**Step 5: Rule Out Security Event**
- [ ] Could this be attack activity?
  - Brute force (failed logins spike)
  - DoS (connection spike)
  - Data exfiltration (data transfer spike)
  - Reconnaissance (DNS query spike)
- [ ] Any correlated alerts?

#### Evidence to Collect
- [ ] Baseline documentation (expected vs. observed)
- [ ] Underlying log events (what caused the spike?)
- [ ] Historical data (has this happened before?)
- [ ] System owner confirmation (legitimate activity?)

#### Escalation Criteria

**Escalate if**:
- ✅ No legitimate business explanation found
- ✅ Deviation is extreme (>2× baseline, not just threshold)
- ✅ Underlying events show attack patterns
- ✅ System owner unaware of cause

**Close as Benign if**:
- ❌ Legitimate business event confirmed
- ❌ Planned activity confirmed (deployment, migration)
- ❌ Historical precedent exists (happens every quarter)

**Update Baseline if**:
- ❌ Business permanently changed (e.g., user base doubled)
- ❌ Recurring false positives (threshold too sensitive)

## 4. Escalation Procedures

### 4.1 When to Escalate

**Always Escalate**:
- ✅ Confirmed True Positive (security event)
- ✅ Any malware detection (no exceptions)
- ✅ Successful compromise (attacker gained access)
- ✅ Data exfiltration (potential or confirmed)
- ✅ Critical system affected (even if benign, inform stakeholders)

**Consider Escalating**:
- ⚠️ Unclear disposition (can't determine TP vs. FP)
- ⚠️ User/system owner unreachable for confirmation
- ⚠️ SLA about to expire (better to escalate than miss SLA)

**Don't Escalate**:
- ❌ Confirmed False Positive (document and close)
- ❌ Benign True Positive (document and close)

### 4.2 How to Escalate

**Step 1: Prepare Brief**
Before escalating, prepare concise summary:
```
ESCALATION BRIEF
-----------------
Alert: [Name]
System: [Hostname]
User: [Username]
Severity: [Critical/High/Medium/Low]
Detected: [DD.MM.YYYY HH:MM]

SITUATION:
[1-2 sentence summary of what happened]

INVESTIGATION FINDINGS:
[Key facts discovered during triage/investigation]

INDICATORS OF COMPROMISE:
- [IOC 1]
- [IOC 2]

RECOMMENDED ACTIONS:
- [Action 1]
- [Action 2]

EVIDENCE COLLECTED:
- [Location of logs, screenshots, etc.]
```

**Step 2: Execute Escalation**

**During Business Hours (08:00-18:00 Mon-Fri)**:
- [ ] Create incident ticket in ticketing system
- [ ] Assign to: SOC Tier 2 (if Tier 1) or Incident Response (if Tier 2)
- [ ] Notify via team chat: "Escalated ticket #INC-XXXX, [1-line summary]"
- [ ] If Critical: Call SOC lead/IR lead directly

**After Hours / Weekends**:
- [ ] Create incident ticket
- [ ] If Critical or High: Page on-call via [on-call system]
- [ ] If Medium: Ticket in queue for next shift
- [ ] If Low: Ticket in queue for next business day

**Contact Information**:
- SOC Tier 2 Escalation: [Email/Chat Channel]
- Incident Response Team: [Email/Chat Channel]
- On-Call Pager: [Phone Number / Paging System]
- SOC Lead (Emergency): [Cell Phone]

## 5. Evidence Collection Guide

### 5.1 What to Collect

**Minimum Evidence** (for any escalation):
- [ ] **Alert details** (screenshot or export)
- [ ] **Timeline** (sequence of events)
- [ ] **Relevant logs** (±2 hours around event)
- [ ] **Context** (user info, asset info, threat intel)

**Additional Evidence** (based on alert type):

**Authentication Events**:
- [ ] Authentication logs (success + failure)
- [ ] User account details (permissions, group membership)
- [ ] Source IP geo-location and reputation

**Malware**:
- [ ] Malware sample (from quarantine)
- [ ] Process execution timeline
- [ ] Network connections (C2 communication)
- [ ] File system changes

**Data Exfiltration**:
- [ ] Network flow logs
- [ ] File access logs
- [ ] Destination analysis

**Lateral Movement**:
- [ ] RDP/SSH connection logs
- [ ] Process execution logs
- [ ] Credential access logs

### 5.2 How to Collect

**SIEM Exports**:
```
1. Run search for relevant time range
2. Export results: CSV or JSON format
3. Save to case folder: /cases/[incident-id]/evidence/
4. Document: What query was run, when, by whom
```

**Log File Downloads**:
```
1. Identify source system
2. SSH/RDP to system
3. Copy relevant log files (preserve timestamps!)
4. Save to case folder
5. Calculate hash (SHA256) for integrity
```

**Screenshots**:
```
1. Capture full screen (include timestamps, URLs)
2. Annotate if needed (highlight relevant sections)
3. Save as PNG with descriptive filename
4. Include in case documentation
```

### 5.3 Chain of Custody

**Document**:
- Who collected evidence
- When (timestamp)
- Where (source system, file path)
- How (method used)
- Hash (for integrity verification)

**Example Log**:
```
EVIDENCE: authentication.log from PROD-DC-01
Collected By: Jane Smith (SOC Tier 1)
Collected: [Approval Date] 15:23:45 UTC
Source: /var/log/auth.log (lines 1000-2000)
Method: SSH download via scp
SHA256: abc123...
Storage: /cases/INC-2847/evidence/auth.log
```

## 6. Common False Positive Scenarios

### 6.1 Failed Authentication Alerts

**Scenario 1: User Password Typo**  
Symptom: 3-5 failed logins, then successful login  
Indicators: Same user, same source IP, normal timing  
Action: Close as benign, no action needed

**Scenario 2: Password Reset**  
Symptom: Failed logins immediately after password change  
Indicators: User using old password, ticket shows recent password reset  
Action: Close as benign, inform user if recurring

**Scenario 3: Saved Credentials**  
Symptom: Repeated failed logins from single workstation  
Indicators: Browser/app has saved old password, automated retries  
Action: Contact user, update saved credentials

---

### 6.2 Baseline Deviation Alerts

**Scenario 1: Maintenance Window**  
Symptom: Activity spike during scheduled maintenance  
Indicators: Change ticket, approved maintenance window  
Action: Close as benign, consider excluding maintenance windows from alerts

**Scenario 2: Business Cycle**  
Symptom: Month-end processing spike  
Indicators: Recurring pattern (every month), expected by system owner  
Action: Update baseline or create exception for month-end period

**Scenario 3: Marketing Campaign**  
Symptom: Website traffic spike  
Indicators: Marketing team launched email campaign  
Action: Close as benign, consider informing SOC of planned campaigns

---

### 6.3 Network Alerts

**Scenario 1: Authorized Security Scanner**  
Symptom: Port scan or vulnerability scan detected  
Indicators: Source is known scanner IP, matches scan schedule  
Action: Whitelist scanner IP for this alert type

**Scenario 2: Backup Jobs**  
Symptom: Large data transfer alert  
Indicators: Destination is backup server, matches backup schedule  
Action: Whitelist backup traffic or adjust threshold

---

### 6.4 Malware Alerts

**Scenario 1: EICAR Test File**  
Symptom: AV detects "malware"  
Indicators: File hash = EICAR test hash, authorized testing  
Action: Close as test file, document testing activity

**Scenario 2: Heuristic False Positive**  
Symptom: Legitimate software flagged as suspicious  
Indicators: Vendor-signed, known application, multiple sources confirm benign  
Action: Whitelist file hash, report FP to vendor
